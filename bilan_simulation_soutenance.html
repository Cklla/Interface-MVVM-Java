3. Réponses à retenir pour le jour J

LiveData vs observable classique

Un observable classique continuerait de notifier le Fragment même après sa destruction, ce qui provoquerait un NullPointerException — on essaie d'accéder à un objet qui n'existe plus en mémoire. Le LiveData est lifecycle-aware : il connaît l'état du Fragment et cesse automatiquement de notifier quand celui-ci est détruit.

Les 3 méthodes de l'Adapter

  • onCreateViewHolder — crée et gonfle le layout XML d'un élément, retourne un ViewHolder vide
  • onBindViewHolder — remplit ce ViewHolder avec les données de l'élément à la position donnée (nom, note, commentaire, avatar)
  • getItemCount — retourne le nombre total d'éléments pour qu'Android sache quand s'arrêter

InstantTaskExecutorRule

Le LiveData utilise normalement le main thread Android pour notifier ses observeurs. Dans un test unitaire, ce thread n'existe pas. InstantTaskExecutorRule le remplace par une exécution synchrone et immédiate, ce qui permet de lire getValue() juste après setValue() dans le test.

Passer à une vraie API

Le fichier à modifier est AppModule.java. C'est lui qui contient l'annotation @Provides qui dit à Hilt : "quand quelqu'un a besoin d'un RestaurantApi, fournis-lui une RestaurantFakeApi". On changerait uniquement ce fichier — le Repository, le ViewModel et le Fragment ne seraient pas touchés. C'est l'un des avantages de l'injection de dépendances combinée à MVVM.

Si l'évaluateur insiste : "Hilt était déjà en place dans le projet livré, je n'ai pas eu de formation sur l'injection de dépendances, mais j'ai compris que AppModule est le point d'entrée qui configure les dépendances."

Qu'est-ce qu'un Adapter ?

L'Adapter est la classe qui fait le lien entre ma liste de données et le RecyclerView — il sait comment créer et remplir chaque ligne d'affichage.
